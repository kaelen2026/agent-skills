# 架构决策记录（ADR）

## 什么是 ADR

ADR（Architecture Decision Record）是一种轻量级文档，用于记录项目中重要的架构和技术决策。

**为什么要记录决策：**

- **避免重复讨论**：新成员不必反复追问"当初为什么选了 X"
- **保留上下文**：决策背后的约束条件和权衡随时间推移容易遗忘
- **支持回溯**：当需要重新评估某个决策时，有据可查
- **促进沟通**：团队成员可以异步了解和讨论架构方向

类比：ADR 就像项目的"判例法"。每次做出重大决定时写下来，后来者可以通过阅读这些记录理解项目演进的脉络。

## 文件命名规范

所有 ADR 存放在 `docs/adr/` 目录下，文件名格式：

```
docs/adr/NNNN-简短标题.md
```

- `NNNN`：四位数字序号，从 `0001` 开始递增
- 标题使用小写英文，单词间用连字符分隔

示例：

```
docs/adr/0001-use-postgresql-over-mongodb.md
docs/adr/0002-adopt-cursor-based-pagination.md
docs/adr/0003-migrate-to-turborepo.md
```

## ADR 模板

```markdown
# NNNN. 决策标题

## 状态

proposed | accepted | deprecated | superseded by [NNNN](NNNN-xxx.md)

## 日期

YYYY-MM-DD

## 背景

描述促使此决策产生的问题或需求。包括：

- 当前面临的具体问题
- 相关的技术约束和业务约束
- 团队现有的技术栈和能力
- 必须满足的需求（功能性 / 非功能性）

## 决策

明确说明做出了什么决定，以及选择该方案的核心理由。

**我们决定：** [具体决策内容]

**核心理由：**

1. 理由一
2. 理由二
3. 理由三

## 后果

### 正面影响

- 具体的正面影响 1
- 具体的正面影响 2

### 负面影响

- 具体的负面影响 1
- 具体的负面影响 2

### 风险

- 风险 1：描述 + 缓解措施
- 风险 2：描述 + 缓解措施

## 备选方案

### 方案 A：[名称]

**优点：**
- 优点 1
- 优点 2

**缺点：**
- 缺点 1
- 缺点 2

**未选择原因：** 简要说明

### 方案 B：[名称]

**优点：**
- 优点 1
- 优点 2

**缺点：**
- 缺点 1
- 缺点 2

**未选择原因：** 简要说明
```

## 状态说明

| 状态 | 含义 |
|------|------|
| `proposed` | 已提出，等待团队讨论和审批 |
| `accepted` | 已通过，团队同意执行 |
| `deprecated` | 已废弃，不再适用但保留历史记录 |
| `superseded by [NNNN]` | 已被新决策替代，附带新 ADR 的链接 |

## ADR 示例

### 示例 1：选用 PostgreSQL 而非 MongoDB

```markdown
# 0001. 选用 PostgreSQL 作为主数据库

## 状态

accepted

## 日期

2026-01-15

## 背景

项目需要持久化存储用户数据、订单数据和商品数据。数据之间存在
复杂的关联关系（用户 -> 订单 -> 商品）。团队评估了关系型数据库
和文档数据库两类方案。

关键约束：
- 数据一致性要求高（涉及支付）
- 需要复杂的联表查询和聚合统计
- 团队成员普遍熟悉 SQL

## 决策

**我们决定：** 使用 PostgreSQL 作为主数据库。

**核心理由：**

1. 强事务支持，满足支付场景的一致性要求
2. 丰富的 JOIN 和聚合能力，适合复杂查询
3. 团队 SQL 经验丰富，学习成本低

## 后果

### 正面影响

- ACID 事务保障数据一致性
- 成熟的生态系统和运维工具
- 强类型 schema 在编译期捕获数据问题

### 负面影响

- 水平扩展不如 MongoDB 灵活
- Schema 变更需要编写迁移脚本

### 风险

- 数据量超预期增长时扩展成本高：可通过读写分离和分区表缓解
- 迁移脚本管理复杂度增加：采用 migration 工具（如 Prisma Migrate）自动化管理

## 备选方案

### 方案 A：MongoDB

**优点：**
- Schema 灵活，适合快速迭代
- 原生水平扩展（sharding）

**缺点：**
- 多文档事务性能较差
- 复杂关联查询需要应用层实现

**未选择原因：** 数据关联复杂且一致性要求高，MongoDB 的优势场景与本项目不匹配。
```

### 示例 2：采用游标分页

```markdown
# 0002. 采用游标分页替代偏移量分页

## 状态

accepted

## 日期

2026-02-10

## 背景

列表接口目前使用 OFFSET/LIMIT 分页。随着数据量增长，深度分页
（如第 1000 页）查询性能显著下降。用户反馈加载缓慢。

数据库分析显示：
- 商品表已超过 500 万条记录
- OFFSET 10000 以上的查询耗时 > 2 秒

## 决策

**我们决定：** 所有列表接口迁移到游标分页（cursor-based pagination）。

**核心理由：**

1. 查询性能与页码深度无关，始终 O(log n)
2. 实时数据场景下不会漏数据或重复数据
3. 行业最佳实践（GitHub、Stripe、Slack 均采用）

## 后果

### 正面影响

- 分页性能稳定，不随数据量增长退化
- 数据一致性更好，无跳页/重复问题

### 负面影响

- 无法直接跳转到任意页码
- 客户端需要改造分页逻辑

### 风险

- 客户端改造工作量：提供 SDK 封装降低接入成本
- 排序字段必须有唯一索引：审查所有排序字段并补充索引

## 备选方案

### 方案 A：优化 OFFSET 分页

**优点：**
- 无需改造客户端
- 实现简单

**缺点：**
- 治标不治本，数据继续增长后问题复发
- 深度分页问题无法根本解决

**未选择原因：** 仅推迟问题而非解决问题，长期来看总成本更高。
```

## 何时编写 ADR

满足以下任一条件时，应当编写 ADR：

| 场景 | 示例 |
|------|------|
| 引入新技术或框架 | 从 REST 迁移到 GraphQL |
| 变更系统架构 | 从单体拆分为微服务 |
| 重大技术选型 | 选择数据库、消息队列、缓存方案 |
| 做出有明显权衡的决定 | 牺牲一致性换取可用性 |
| 废弃或替换现有方案 | 替换认证框架 |
| 建立新的技术规范 | 统一 API 版本策略 |

**不需要编写 ADR 的情况：**

- 日常的 bug 修复
- 小规模的代码重构
- 依赖版本升级（除非是主版本破坏性升级）
- 团队已有共识且无争议的常规做法
