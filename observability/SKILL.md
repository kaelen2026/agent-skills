# 可观测性技能

## 何时激活

以下场景出现时自动激活本技能：

- 设置监控系统或仪表盘
- 配置告警规则
- 实现结构化日志
- 集成分布式追踪
- 设计健康检查端点
- 定义 SLO/SLI/SLA

## 工作流程

### 第一步：定义 SLO/SLI

在编写任何代码之前，先明确服务的可靠性目标。

1. 确定关键用户旅程
2. 为每个旅程定义 SLI（服务等级指标）
3. 设定 SLO（服务等级目标）
4. 计算错误预算

参考：[监控与告警](references/monitoring-alerting.md)

### 第二步：配置日志

结构化日志是可观测性的基础。

1. 选择日志库（推荐 pino）
2. 统一 JSON 格式输出
3. 实现请求级别的关联 ID（requestId）
4. 配置敏感数据过滤
5. 设置日志聚合与轮转策略

参考：[日志](references/logging.md)

### 第三步：设置监控

基于 SLO 构建监控体系。

1. 收集 RED 指标（Rate、Errors、Duration）
2. 收集 USE 指标（Utilization、Saturation、Errors）
3. 搭建仪表盘：总览 → 服务 → 端点逐层下钻
4. 实现健康检查端点

参考：[监控与告警](references/monitoring-alerting.md)、[追踪](references/tracing.md)

### 第四步：配置告警

告警必须可操作，避免告警疲劳。

1. 按严重程度分级（P1-P4）
2. 配置路由与升级策略
3. 为每条告警编写运行手册
4. 定期审查告警有效性

参考：[监控与告警](references/monitoring-alerting.md)

## 可观测性三大支柱

| 支柱 | 用途 | 核心问题 |
|------|------|----------|
| 日志（Logs） | 记录离散事件 | 发生了什么？ |
| 指标（Metrics） | 聚合数值度量 | 系统表现如何？ |
| 追踪（Traces） | 跨服务请求链路 | 请求经过了哪些环节？ |

三者缺一不可。日志提供上下文，指标提供趋势，追踪提供因果链。

## 审查清单

实施可观测性后，逐项确认：

- [ ] 所有服务使用结构化 JSON 日志
- [ ] requestId 在服务间正确传播
- [ ] 敏感数据（密码、令牌、PII）已从日志中过滤
- [ ] RED 指标（Rate、Errors、Duration）已收集
- [ ] USE 指标（Utilization、Saturation、Errors）已收集
- [ ] SLO/SLI 已定义且有对应仪表盘
- [ ] 错误预算已计算并可追踪
- [ ] 告警规则已配置且均可操作
- [ ] 每条告警有对应的运行手册
- [ ] 健康检查端点（`/health`、`/ready`）已实现
- [ ] 分布式追踪已集成，span 属性完整
- [ ] 日志轮转与保留策略已配置
- [ ] 仪表盘支持总览 → 服务 → 端点的逐层下钻
- [ ] 优雅降级模式已实现
