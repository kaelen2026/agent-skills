# Grill-Me 追问

就像律师交叉询问一样——不是为了刁难，而是确保每个关键决策都经过深思熟虑，没有盲区。

## When to Activate

- 用户说"追问我"、"grill me"、"帮我想清楚"
- 其他技能需要深挖需求时显式调用（如 write-a-prd 步骤 3）
- PRD 评审、设计评审、架构讨论中存在模糊点
- 用户描述了一个想法但细节不清

## Workflow

### 1. 理解上下文

确认追问对象——PRD、设计方案、架构提案、模糊需求等。

- 快速扫描已有信息（文档、对话历史、代码）
- 识别追问对象的类型和当前成熟度
- 向用户确认："我将对 [X] 进行追问，确认吗？"

### 2. 构建决策树

识别所有需要决策的分支点：

- **边界情况** — "如果 X 发生了怎么办？"
- **优先级冲突** — "A 和 B 冲突时选哪个？"
- **范围模糊** — "这个算 MVP 还是后续迭代？"
- **隐含假设** — "你假设了 X，但如果不是呢？"
- **技术选型** — "为什么选 A 而不是 B？"
- **用户视角** — "用户在这一步期望看到什么？"

按影响力排序：影响架构 > 影响数据模型 > 影响 API > 影响 UI > 影响文案。

### 3. 逐分支追问

循环追问，直到所有分支都有明确答案或用户说"够了"。

**每轮规则**：
- 最多 3 个问题，从最高影响力分支开始
- 用 MoSCoW（Must/Should/Could/Won't）标记优先级争议
- 每个问题给出选项或示例，降低用户回答成本
- 标记分支状态：✅ 已解决 / ❓ 待确认 / ⏭️ 延后

**结束条件**：
- 所有高影响力分支已解决
- 用户明确说"够了"或"继续下一步"

### 4. 输出决策摘要

汇总所有决策为结构化清单：

```
## 决策摘要

| # | 决策点 | 选择 | 理由 | 优先级 |
|---|--------|------|------|--------|
| 1 | ...    | ...  | ...  | Must   |

### 延后项
- [ ] ...（标记为后续迭代处理）
```

此摘要可直接被其他技能（write-a-prd、planning、api-design 等）消费。

## Design Review

追问完成后自动检查：

- [ ] 所有高影响力分支都有明确答案
- [ ] 优先级冲突已用 MoSCoW 标记
- [ ] 决策摘要包含选择和理由
- [ ] 延后项已明确标记
- [ ] 无遗留的 ❓ 待确认分支（或已获用户确认延后）
