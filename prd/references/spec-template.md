## 补充模板

> 以下模板可按需整合到实施规格中，用于更精确的工程规划。

---

### 切片估算指南

> 为每个 Phase 的工作量提供标准化估算。用 T-shirt sizing 快速对齐团队预期。

#### T-shirt 尺码定义

| 尺码 | 工时范围 | 典型特征 | 示例 |
|------|---------|---------|------|
| **S** | 1-4 小时 | 单文件修改，无依赖，模式已知 | 新增一个 CRUD 端点 |
| **M** | 4-16 小时 | 2-4 文件，有简单依赖，模式需小幅调整 | 新增带权限的 API + 前端表单 |
| **L** | 16-40 小时 | 5-8 文件，跨模块依赖，需要设计决策 | 新增完整功能模块 + 测试 |
| **XL** | 40-80 小时 | 8+ 文件，架构变更，需拆分为子切片 | 认证系统重构 |

> **规则**: 如果估算为 XL，必须拆分为更小的切片。单个切片不应超过 L。

#### 复杂度因子

估算时考虑以下因子，每命中一项工时 ×1.3:

- [ ] 涉及数据迁移
- [ ] 需要第三方 API 集成
- [ ] 包含实时/WebSocket 通信
- [ ] 需要处理并发/竞态条件
- [ ] 涉及金融计算或精度要求
- [ ] 需要跨时区处理

#### 不确定性乘数

| 团队经验 | 乘数 | 适用场景 |
|---------|------|---------|
| 熟悉领域 | ×1.0 | 团队做过类似功能 |
| 部分熟悉 | ×1.5 | 用过技术栈但没做过此类功能 |
| 全新领域 | ×2.0 | 新技术栈或全新业务领域 |

#### 估算模板

```markdown
### Phase {N} 估算

| 维度 | 评估 |
|------|------|
| 基础尺码 | {S / M / L} |
| 复杂度因子 | {命中 N 项, ×1.3^N 或 ×1.3 每项} |
| 不确定性 | {×1.0 / ×1.5 / ×2.0} |
| 最终估算 | {基础工时 × 复杂度 × 不确定性 = N 小时} |
| 信心水平 | {高 / 中 / 低} |
```

### 依赖分析模板

> 在「切片总览」之后插入。识别阻塞风险，提前准备替代方案。

```markdown
## 依赖分析

### 依赖清单

| # | 依赖项 | 类型 | 负责人 | 风险等级 | 替代方案 |
|---|--------|------|--------|---------|---------|
| D1 | {内部服务: 用户认证 API} | 内部 | {团队/人} | {低/中/高} | {Mock 接口先行开发} |
| D2 | {外部服务: 支付网关} | 外部 | {供应商} | {低/中/高} | {沙盒环境 + 本地模拟} |
```

### 测试策略章节

> 每个 Phase 的验收标准之后插入。确保测试覆盖有明确目标。

```markdown
## 测试策略

### 覆盖率目标

| 测试层级 | 覆盖率目标 | 关注重点 |
|---------|-----------|---------|
| 单元测试 | {≥ 80%} | {核心业务逻辑、工具函数、数据转换} |
| 集成测试 | {≥ 60%} | {API 端点、数据库交互、服务间调用} |
| E2E 测试 | {关键路径 100%} | {用户核心流程、支付流程、注册登录} |

### 各 Phase 测试计划

#### Phase {N} 测试

**单元测试**:
- [ ] {模块 A: 核心计算逻辑 — 正常/边界/异常}
- [ ] {模块 B: 数据校验 — 合法/非法输入}
- [ ] {模块 C: 状态转换 — 所有合法路径}

**集成测试**:
- [ ] {API 端点: 请求/响应/错误码}
- [ ] {数据库: CRUD + 事务回滚}
- [ ] {外部服务: Mock 响应 + 超时处理}

**E2E 测试**:
- [ ] {关键路径 1: {用户操作流程描述}}
- [ ] {关键路径 2: {用户操作流程描述}}

### 测试数据需求

| 数据类型 | 来源 | 数量 | 准备方式 |
|---------|------|------|---------|
| {用户数据} | {Seed / Factory} | {100 条} | {测试 Fixture} |
| {交易数据} | {生成器} | {1000 条} | {Factory 函数} |
| {边界数据} | {手动构造} | {20 条} | {硬编码在测试中} |

### 关键边界条件

- {空数据 / 空列表处理}
- {并发写入同一资源}
- {网络超时 / 部分失败}
- {数据量极大时的分页/流式处理}
- {时区边界 (UTC 午夜前后)}
```

### 回滚计划

> 在「开放问题」之前插入。发布失败时的应急方案，提前写好才不会临时慌乱。

```markdown
## 回滚计划

### 触发条件

以下任一情况出现时触发回滚:

- [ ] {P0 错误: 核心流程完全不可用}
- [ ] {错误率 > {N}% (正常基线的 {M} 倍)}
- [ ] {API 响应时间 P95 > {N}ms}
- [ ] {数据一致性异常}

### 回滚步骤

| 步骤 | 操作 | 负责人 | 预计耗时 |
|------|------|--------|---------|
| 1 | {切换流量到旧版本 (蓝绿/金丝雀回退)} | {运维} | {< 5 分钟} |
| 2 | {暂停异步任务队列} | {后端} | {< 2 分钟} |
| 3 | {确认旧版本服务正常} | {QA} | {< 10 分钟} |
| 4 | {通知相关方} | {PM} | {< 5 分钟} |

### 数据恢复

| 场景 | 恢复策略 | 数据损失范围 |
|------|---------|------------|
| {Schema 迁移已执行} | {执行 down migration} | {无损失} |
| {新数据已写入新表} | {保留新数据，标记待迁移} | {需人工处理} |
| {旧数据已被修改} | {从备份恢复 + 增量重放} | {取决于备份频率} |

### 沟通计划

| 对象 | 渠道 | 时机 | 内容模板 |
|------|------|------|---------|
| {工程团队} | {Slack #incidents} | {立即} | {回滚已触发，原因: {X}} |
| {产品团队} | {邮件} | {回滚完成后} | {功能暂时下线，预计 {N} 恢复} |
| {用户} | {状态页} | {影响用户时} | {部分功能维护中} |
```

### 监控指标

> 在「回滚计划」之后插入。上线后通过指标验证功能健康度。

```markdown
## 监控指标

| 指标名称 | 类型 | 阈值 | 告警级别 | 说明 |
|---------|------|------|---------|------|
| {功能使用次数} | Counter | {N/min 基线} | INFO | {追踪采用率} |
| {转化率} | Gauge | {< {N}% 告警} | WARNING | {核心业务指标} |
| {用户流失点} | Counter | {> {N}/hour} | WARNING | {漏斗分析} |

### 技术指标

| 指标名称 | 类型 | 阈值 | 告警级别 | 说明 |
|---------|------|------|---------|------|

| {错误率} | Gauge | {> {N}%} | CRITICAL | {立即触发 on-call} |
| {队列积压} | Gauge | {> {N} 条} | WARNING | {Worker 处理能力} |
| {数据库连接使用率} | Gauge | {> 80%} | WARNING | {资源瓶颈预警} |
| {缓存命中率} | Gauge | {< {N}%} | INFO | {缓存策略有效性} |

### Dashboard 规划

| Dashboard | 包含指标 | 受众 | 刷新频率 |
|-----------|---------|------|---------|
```
