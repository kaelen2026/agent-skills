# 需求收集指南

> 用于 `prd` 技能 Phase 0（PRD 编写前）的需求收集工具箱。好的 PRD 始于好的需求收集。

---

## 用户访谈问题清单

> 按主题分类的访谈问题。根据项目类型选择相关问题，每次访谈选 8-12 题。

### 痛点挖掘

1. 目前这个流程中，最让你头疼的环节是什么？
2. 上一次因为这个问题浪费了多少时间？
3. 你有没有试过什么替代方案或临时解决办法？效果如何？
4. 如果这个问题不解决，最坏的结果是什么？

### 当前工作流

1. 能不能带我走一遍你现在的完整操作流程？（从头到尾）
2. 这个流程中哪些步骤是重复性的 / 可以自动化的？
3. 你在不同工具之间切换的频率有多高？主要切换哪些？
4. 团队中不同角色的流程有什么差异？

### 期望与优先级

1. 如果有一个魔法按钮能解决一个问题，你最想解决哪个？
2. 你对「好用」的定义是什么？能举个你觉得体验好的产品例子吗？
3. 速度、稳定性、功能丰富度 — 你怎么排序？
4. 你愿意为这个功能改变现有的工作习惯吗？改变多少？

### 边界与异常

1. 有没有遇到过数据特别多 / 特别少的极端情况？怎么处理的？
2. 多人同时操作同一个东西时会发生什么？
3. 系统出错时你期望看到什么？目前的错误提示够清楚吗？
4. 有没有「理论上不应该但实际会发生」的奇怪场景？

### 访谈技巧提示

- 用「能给我举个例子吗？」追问模糊回答
- 观察用户实际操作，而不只是听描述
- 记录用户的原话（verbatim），不要提前翻译成技术语言
- 注意用户**没说**的东西 — 可能是习以为常的痛点

## 利益相关者映射

> 识别所有利益相关者，制定差异化沟通策略。

### 映射模板

| 姓名/角色 | 利益程度 | 影响力 | 象限 | 参与策略 |
|----------|---------|--------|------|---------|
| {PM / 产品经理} | {高} | {高} | 密切合作 | {每周同步，决策共识} |
| {工程负责人} | {高} | {高} | 密切合作 | {技术方案评审} |
| {设计师} | {高} | {中} | 保持满意 | {设计评审，原型确认} |
| {客服团队} | {中} | {低} | 保持知情 | {月度更新，反馈收集} |
| {法务/合规} | {低} | {高} | 满足要求 | {合规审查节点} |
| {最终用户代表} | {高} | {低} | 保持知情 | {Beta 测试，访谈} |

### 象限说明

```
            高影响力
              │
   满足要求   │   密切合作
   (Satisfy)  │   (Collaborate)
              │
 低利益 ──────┼────── 高利益
              │
   最低关注   │   保持知情
   (Monitor)  │   (Inform)
              │
            低影响力
```

- **密切合作**: 高利益 + 高影响力 → 深度参与决策
- **保持满意**: 高利益 + 低影响力 → 定期更新，收集反馈
- **满足要求**: 低利益 + 高影响力 → 关键节点审批
- **最低关注**: 低利益 + 低影响力 → 按需通知

## 需求优先级矩阵

### MoSCoW 分类法

| 类别 | 含义 | 判断标准 | 占比建议 |
|------|------|---------|---------|
| **Must** | 没有就不能发布 | 核心价值、合规要求、安全底线 | 60% |
| **Should** | 重要但有变通方案 | 显著提升体验，但手动可替代 | 20% |
| **Could** | 锦上添花 | 有则更好，无也不影响核心流程 | 15% |
| **Won't** | 本次不做 | 明确排除，避免范围蔓延 | 5% (记录但不做) |

### RICE 评分模板

> 用于在 Must/Should 内部进一步排序。

| 需求 | Reach | Impact | Confidence | Effort | RICE 得分 |
|------|-------|--------|-----------|--------|----------|
| {需求 A} | {每月影响用户数} | {0.25/0.5/1/2/3} | {50%/80%/100%} | {人月} | {R×I×C÷E} |
| {需求 B} | {用户数} | {影响度} | {信心} | {人月} | {得分} |
| {需求 C} | {用户数} | {影响度} | {信心} | {人月} | {得分} |

#### RICE 评分说明

- **Reach**: 每月/每季度受影响的用户数或事件数
- **Impact**: 对单个用户的影响程度
  - 3 = 巨大 (解决核心痛点)
  - 2 = 高 (显著改善)
  - 1 = 中 (有帮助)
  - 0.5 = 低 (微小改善)
  - 0.25 = 最低 (几乎无感)
- **Confidence**: 对以上估算的信心
  - 100% = 有数据支撑
  - 80% = 有定性反馈
  - 50% = 直觉判断
- **Effort**: 完成所需的人月数 (越大得分越低)

## 需求验证检查清单

> PRD 完成后、进入实施规格之前，用此清单验证需求质量。

### 完整性

- [ ] 所有用户角色都已覆盖
- [ ] 正常流程和异常流程都有描述
- [ ] 非功能性需求已明确（性能、安全、可用性）
- [ ] 数据需求已定义（输入、输出、存储、保留策略）
- [ ] 边界条件和极端情况已记录

### 一致性

- [ ] 需求之间没有互相矛盾
- [ ] 术语定义一致（同一概念同一名称）
- [ ] 优先级排序与业务目标一致
- [ ] 与现有系统功能没有冲突

### 可行性

- [ ] 技术方案已初步验证（或标记为需要 Spike）
- [ ] 在当前团队能力范围内
- [ ] 在预算和时间约束内可完成
- [ ] 第三方依赖已确认可用性和 SLA

### 可测试性

- [ ] 每个需求都有明确的验收标准
- [ ] 验收标准是可观察的外部行为（非实现细节）
- [ ] 可以编写自动化测试来验证
- [ ] 性能指标有具体数值（不是「快」「流畅」等模糊描述）

### 最终确认

- [ ] 所有 Must Have 需求已获利益相关者签字确认
- [ ] Won't Have 列表已明确，防止范围蔓延
- [ ] 开放问题列表已更新，有明确的解决时间线
- [ ] 文档已分享给所有相关方
