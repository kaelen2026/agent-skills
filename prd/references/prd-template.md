# PRD Template

> 用于 `prd` 技能 Phase 1 的 PRD 模板。

```markdown
## 问题描述

{用非技术语言描述用户痛点。一段话，任何人都能理解。}

## 目标

{这个功能要达成什么？用 1-3 个可衡量的目标。}

## 目标用户

{谁会使用这个功能？列出角色和使用场景。}

## 用户故事

### Must Have

- [ ] 作为 {角色}，我想要 {行为}，以便 {价值}
  - 前置条件: {条件}
  - 期望结果: {详细描述用户看到/体验到的}
  - 边界情况: {异常情况及处理方式}

### Should Have

- [ ] ...

### Could Have

- [ ] ...

### Won't Have (本次不做)

- ...

## 模块设计

### 模块 1: {名称}

- **接口**: {输入 → 输出，尽量简单}
- **隐藏复杂性**: {模块内部处理的复杂逻辑}
- **依赖**: {依赖的其他模块}

### 模块 2: {名称}

- **接口**: ...
- **隐藏复杂性**: ...
- **依赖**: ...

## 架构决策

| 决策 | 选项 | 选择 | 理由 |
|------|------|------|------|
| {决策点} | A: ... / B: ... | {选择} | {为什么} |

## Schema 变更

{新增/修改的数据模型。只描述字段和关系，不含具体 SQL。}

## API 契约

{新增/修改的 API 端点。包含路径、方法、请求/响应结构、错误码。}

## 验收标准

- [ ] {描述可观察的外部行为，非实现细节}
- [ ] {用户执行 X → 系统响应 Y}
- [ ] {错误场景: 当 Z 发生时 → 用户看到 W}

## 未解决问题

- {需要进一步讨论的开放问题}

## 参考

- {相关文档、设计稿、竞品链接}
```

## 补充模板

> 以下模板可按需整合到 PRD 中，用于更完整的需求分析。

### 竞品分析模板

> 在 PRD 的「参考」章节之前插入。了解市场现有方案，明确差异化定位。

**使用方式**: 列出 3-5 个直接/间接竞品，逐项评估。重点关注他们**没做好的地方** — 那就是我们的机会。

```markdown
## 竞品分析

### 市场定位

{一句话描述本产品在市场中的定位，与竞品的核心差异。}

### 竞品对比

| 竞品 | 核心优势 | 主要劣势 | 我们的差异化 |
|------|---------|---------|-------------|
| {竞品 A} | {他们做得好的 2-3 点} | {他们的短板} | {我们如何利用这个差距} |
| {竞品 B} | {优势} | {劣势} | {差异化策略} |
| {竞品 C} | {优势} | {劣势} | {差异化策略} |

### 功能对比矩阵

| 功能点 | 本产品 | {竞品 A} | {竞品 B} | {竞品 C} |
|--------|--------|---------|---------|---------|
| {核心功能 1} | ✅ 计划中 | ✅ 已有 | ❌ 缺失 | ⚠️ 部分支持 |
| {核心功能 2} | ✅ 计划中 | ❌ 缺失 | ✅ 已有 | ✅ 已有 |
| {差异化功能} | ✅ 计划中 | ❌ 缺失 | ❌ 缺失 | ❌ 缺失 |

### 关键洞察

- {从竞品分析中得出的 2-3 条关键结论}
- {用户从竞品迁移的动力是什么？}
```

### 技术约束章节

> 在「架构决策」之后插入。提前明确技术边界，避免设计阶段做出不可行的决策。

```markdown
## 技术约束

### 基础设施限制

| 约束项 | 当前限制 | 影响范围 | 应对策略 |
|--------|---------|---------|---------|
| {数据库连接池} | {最大 100 连接} | {高并发场景} | {连接复用 + 队列} |
| {文件存储} | {单文件最大 {N}MB} | {上传功能} | {分片上传} |
| {内存限制} | {容器 {N}GB RAM} | {批处理任务} | {分批处理 + 流式} |

### 浏览器兼容性

| 浏览器 | 最低版本 | 备注 |
|--------|---------|------|
| Chrome | {最近 2 个大版本} | {主要目标} |
| Safari | {版本号} | {iOS 用户} |
| Firefox | {版本号} | {次要支持} |

### 性能预算

| 指标 | 目标值 | 当前值 | 差距 |
|------|--------|--------|------|
| 首屏加载 (LCP) | {< 2.5s} | {当前值} | {差距} |
| 交互响应 (INP) | {< 200ms} | {当前值} | {差距} |
| API 响应时间 (P95) | {< 500ms} | {当前值} | {差距} |
| JS 包大小 | {< {N}KB gzipped} | {当前值} | {差距} |

### 安全要求

- [ ] {数据加密: 传输中 TLS 1.2+，静态 AES-256}
- [ ] {认证: {JWT / Session / OAuth2}}
- [ ] {授权: {RBAC / ABAC / 行级安全}}
- [ ] {合规: {GDPR / SOC2 / 其他}}
- [ ] {数据保留: {保留策略及自动清理周期}}
```

### 数据流图模板

> 在「模块设计」之后插入。可视化整体数据流向，帮助团队对齐系统交互理解。

```markdown
## 数据流图

### 核心数据流

\```mermaid
flowchart LR
    U[👤 用户] -->|操作| FE[前端应用]
    FE -->|API 请求| GW[API 网关]
    GW -->|认证校验| AUTH[认证服务]
    GW -->|业务请求| API[业务 API]
    API -->|读写| DB[(数据库)]
    API -->|缓存| CACHE[(缓存层)]
    API -->|异步任务| QUEUE[消息队列]
    QUEUE -->|消费| WORKER[后台任务]
    WORKER -->|写入| DB
    API -->|调用| EXT[外部服务]
    EXT -->|回调/响应| API
    API -->|事件| LOG[日志/监控]
\```

### 数据流说明

| 流向 | 协议 | 数据格式 | 频率 | 备注 |
|------|------|---------|------|------|
| {用户 → 前端} | {HTTPS} | {JSON} | {用户触发} | {前端校验} |
| {前端 → API} | {HTTPS REST/gRPC} | {JSON/Protobuf} | {每次操作} | {鉴权 Token} |
| {API → 数据库} | {TCP/TLS} | {SQL} | {每次请求} | {连接池} |
| {API → 外部服务} | {HTTPS} | {JSON} | {按需} | {超时 {N}s，重试 {N} 次} |
| {队列 → Worker} | {AMQP/SQS} | {JSON} | {异步} | {死信队列兜底} |
```

> **注意**: Mermaid 图中的反引号在实际使用时去掉转义符 `\`。

### 里程碑规划

> 在 PRD 末尾、「未解决问题」之前插入。将功能拆解为可交付的里程碑，便于追踪进度。

```markdown
## 里程碑规划

### 总览

| # | 里程碑 | 交付物 | 目标日期 | 成功标准 |
|---|--------|--------|---------|---------|
| M1 | {基础能力} | {核心 API + 最小 UI} | {YYYY-MM-DD} | {端到端流程跑通} |
| M2 | {核心体验} | {完整 UI + 错误处理} | {YYYY-MM-DD} | {验收标准全部通过} |
| M3 | {扩展功能} | {高级功能 + 性能优化} | {YYYY-MM-DD} | {性能指标达标} |
| M4 | {发布准备} | {文档 + 监控 + 灰度} | {YYYY-MM-DD} | {灰度无 P0/P1 问题} |

### 里程碑详情

#### M1: {基础能力} — 目标: {YYYY-MM-DD}

**交付物**:
- [ ] {交付项 1: 具体描述}
- [ ] {交付项 2: 具体描述}

**成功标准**:
- [ ] {可验证的标准 1}
- [ ] {可验证的标准 2}

**前置依赖**:
- {依赖 1: 状态}
- {依赖 2: 状态}

### 关键路径

{哪些里程碑是串行的？哪些可以并行？标注阻塞风险。}
```

### 风险评估矩阵

> 在「里程碑规划」之后插入。提前识别风险，准备缓解方案。

```markdown
## 风险评估

### 风险矩阵

| # | 风险描述 | 概率 (P) | 影响 (I) | 严重度 (P×I) | 缓解策略 |
|---|---------|---------|---------|-------------|---------|
| R1 | {技术风险: 如第三方 API 不稳定} | {1-5} | {1-5} | {计算值} | {备用方案 + 熔断机制} |
| R2 | {人员风险: 如关键成员离职} | {1-5} | {1-5} | {计算值} | {知识共享 + 文档化} |
| R3 | {需求风险: 如范围蔓延} | {1-5} | {1-5} | {计算值} | {严格变更审批流程} |
| R4 | {时间风险: 如依赖延迟} | {1-5} | {1-5} | {计算值} | {并行开发 + 缓冲期} |
| R5 | {安全风险: 如数据泄露} | {1-5} | {1-5} | {计算值} | {安全审计 + 加密} |

### 评分说明

- **概率**: 1=极低, 2=低, 3=中, 4=高, 5=极高
- **影响**: 1=可忽略, 2=轻微, 3=中等, 4=严重, 5=灾难性
- **严重度阈值**: ≥15 立即处理, 10-14 密切关注, <10 定期回顾

### 应急预案

> 针对严重度 ≥15 的风险，必须有详细应急预案。

#### {风险 R1} 应急预案

- **触发条件**: {什么情况下启动应急？}
- **响应步骤**:
  1. {第一步: 立即行动}
  2. {第二步: 短期缓解}
  3. {第三步: 长期修复}
- **负责人**: {谁来执行？}
- **沟通计划**: {通知谁？通过什么渠道？}
```
