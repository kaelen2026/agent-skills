### 4.1 并行执行策略详细指南

**识别可并行任务的判断标准**:

| 判断条件 | 可并行 | 不可并行 |
|----------|--------|----------|
| I/O 独立性 | 操作不同文件/端点 | 操作同一文件/共享状态 |
| 数据依赖 | 无数据流依赖关系 | B 的输入是 A 的输出 |
| Schema 依赖 | 各自独立的表/类型 | 共享 DB 表或类型定义 |
| 测试隔离 | 测试数据互不干扰 | 共用测试 fixture 或种子数据 |

**代理分配模式**:

| 任务类型 | 推荐代理 | 原因 |
|----------|----------|------|
| 数据模型 + 迁移 | tdd-guide | 需要先写测试验证 schema |
| API 端点实现 | tdd-guide | TDD 驱动，确保契约正确 |
| 前端组件 | code-reviewer (审查) | 实现后立即审查 |
| 安全相关变更 | security-reviewer | 需要安全视角审查 |
| 基础设施/部署 | planner (规划) | 需要跨系统协调 |

**协调同步点**:

在以下节点，所有并行分支必须汇合后才能继续：

| 同步点 | 触发条件 | 验证方法 |
|--------|----------|----------|
| Schema 合并 | 所有 DB 迁移完成 | `{迁移命令} && {测试命令}` |
| API 集成 | 前后端各自完成 | {集成测试命令} |
| 部署前 | 所有代码合并到主分支 | `{构建命令} && {全量测试命令}` |

## 7. 检查点定义

### 7.1 检查点标准

每个阶段结束时必须达到以下状态才算「检查点通过」：

| 条件 | 验证方式 | 必须 |
|------|----------|------|
| 代码可编译 | `{构建命令, 如: npm run build}` | 是 |
| 该阶段测试全部通过 | `{测试命令, 如: npm test -- --grep "Phase{N}"}` | 是 |
| 无 lint 错误 | `{lint 命令, 如: npm run lint}` | 是 |
| 可独立部署（不破坏现有功能） | `{集成测试 / smoke test 命令}` | 是 |
| 无未提交的临时代码 | `git status` 无未追踪文件 | 是 |

### 7.2 各阶段验证命令

| 阶段 | 检查点验证命令 |
|------|---------------|
| Phase 1 | `{npm run build && npm test -- --grep "Phase1" && npm run lint}` |
| Phase 2 | `{npm run build && npm test -- --grep "Phase[12]" && npm run lint}` |
| Phase 3 | `{npm run build && npm test && npm run lint}` |

### 7.3 状态保存与恢复

**保存检查点**:

```bash
# 每个检查点通过后立即 commit
git add -A
git commit -m "{type}: {阶段描述} (checkpoint Phase {N})"
```

**恢复到检查点**:

```bash
# 查看检查点记录
git log --oneline --grep="checkpoint"

# 恢复到指定检查点（创建新分支，保留当前进度）
git branch backup-current
git reset --hard {checkpoint-commit-hash}
```

### 7.4 提交时机

| 时机 | 是否提交 | 说明 |
|------|----------|------|
| 检查点通过后 | 是（必须） | 作为恢复点和进度记录 |
| 阶段内任务完成 | 可选 | 大型阶段可设中间检查点 |
| 阶段进行中 | 否 | 未通过检查点的代码不提交 |
| 调试/实验代码 | 否 | 使用 git stash 暂存 |

## 8. 回滚触发条件

### 8.1 自动触发条件

| 触发条件 | 阈值 | 检测方式 | 回滚范围 |
|----------|------|----------|----------|
| 测试失败率 | {> 10% 的测试失败} | CI 管道 | 回滚到上一个检查点 |
| 构建失败 | 任何编译错误 | CI 管道 | 回滚到上一个检查点 |
| 性能回归 | {P95 延迟增加 > 50%} | 性能监控 | 回滚部署 + 调查 |
| 错误率飙升 | {5xx 错误率 > 1%} | APM 告警 | 立即回滚部署 |
| 内存泄漏 | {内存使用持续增长 > 30 分钟} | 基础设施监控 | 回滚部署 + 调查 |

### 8.2 手动触发条件

| 触发条件 | 判断者 | 处理方式 |
|----------|--------|----------|
| 需求变更（影响已实现部分） | 产品负责人 | 暂停开发, 重新评估规格书 |
| 利益相关者否决 | 决策者 | 回滚到否决前的检查点 |
| 阻塞性外部依赖 | 技术负责人 | 暂停受阻分支, 切换到替代方案 |
| 安全漏洞发现 | 安全审查者 | 立即回滚 + 修复后重新实施 |

**代码回滚**:

```bash

git commit -m "revert: 回滚 Phase {N} (原因: {原因})"

# 方式 2: 使用 feature flag 禁用功能
# 在代码中: if (!featureFlags.{FEATURE_NAME}) return fallback;
```

**数据库回滚**:

| 阶段 | 回滚方式 | 命令 |
|------|----------|------|
| 迁移已执行, 无数据写入 | 反向迁移 | `{迁移回滚命令, 如: npx prisma migrate reset}` |
| 迁移已执行, 有数据写入 | 反向迁移 + 数据修复脚本 | `{自定义回滚脚本路径}` |
