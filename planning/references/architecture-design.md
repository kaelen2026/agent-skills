# 架构设计参考

## 1. 技术选型框架

评估候选技术时，使用以下维度进行打分（1-5 分）：

| 维度 | 说明 | 权重参考 |
|------|------|----------|
| 成熟度 | 稳定版本、生产实绩、社区规模 | 高 |
| 生态系统 | 插件/库丰富度、第三方集成 | 中-高 |
| 性能 | 基准测试数据、内存/CPU 开销 | 视场景 |
| 学习曲线 | 团队现有经验、文档质量、入门难度 | 中 |
| 维护成本 | 升级频率、破坏性变更历史、长期支持 | 高 |
| 类型安全 | TypeScript 支持、类型定义质量 | 中-高 |


## 3. C4 模型层级

使用 C4 模型从高到低逐层描述系统架构：

### Level 1: System Context（系统上下文）

系统与外部用户、外部系统的关系。回答"系统在更大的环境中如何定位？"

```
[用户] → [我们的系统] → [外部服务 A]
                      → [外部服务 B]
```

描述内容：
- 系统的主要用户/角色
- 系统依赖的外部服务
- 数据流方向

### Level 2: Container（容器）

系统内部的主要技术组件（Web App、API、数据库等）。回答"系统由哪些可部署的单元组成？"

```
[Web App (Next.js)] → [API Server (Node.js)] → [Database (PostgreSQL)]
                                              → [Cache (Redis)]
                                              → [Queue (BullMQ)]
```

描述内容：
- 每个容器的技术选型
- 容器之间的通信协议（REST、gRPC、WebSocket）
- 数据存储方式

### Level 3: Component（组件）

单个容器内部的主要模块/层。回答"容器内部如何组织？"

```
API Server:
├── Routes (路由层)
├── Controllers (控制层)
├── Services (业务逻辑层)
├── Repositories (数据访问层)
└── Middleware (中间件)
```

描述内容：
- 模块职责划分
- 模块之间的依赖方向（单向依赖）
- 关键接口定义

### Level 4: Code（代码）

仅在需要解释复杂逻辑时使用。通常通过代码本身表达。

## 4. 常见架构模式选择指南

### Monolith vs Microservices

| 维度 | Monolith | Microservices |
|------|----------|---------------|
| 适用阶段 | 早期/中期产品、团队 < 10 人 | 成熟产品、多团队独立部署 |
| 复杂度 | 低（单一代码库） | 高（分布式系统复杂度） |
| 部署 | 整体部署 | 独立部署 |
| 数据一致性 | 事务保证 | 最终一致性 |
| 推荐起点 | **默认选择** | 当 monolith 遇到明确瓶颈时才迁移 |

**原则**: Start with monolith, extract when needed。过早拆分微服务是常见的过度工程。

### SSR vs SPA vs SSG

| 维度 | SSR (Next.js) | SPA (React/Vite) | SSG (Astro/Next.js) |
|------|--------------|-------------------|---------------------|
| SEO | 优秀 | 需额外处理 | 优秀 |
| 首屏性能 | 好（服务端渲染） | 依赖 bundle 大小 | 最佳（预生成） |
| 交互性 | 完整 | 完整 | 有限（需 islands） |
| 服务器成本 | 需要 | 仅静态托管 | 仅静态托管（构建时） |
| 适用场景 | 动态内容 + SEO | 后台管理/工具 | 内容站点/博客/文档 |

### SQL vs NoSQL

| 维度 | SQL (PostgreSQL) | NoSQL (MongoDB/DynamoDB) |
|------|-----------------|--------------------------|
| 数据关系 | 强关系、JOIN 查询 | 弱关系、嵌套文档 |
| 一致性 | ACID 事务 | 最终一致性（多数） |
| 查询灵活性 | 任意查询 | 按预定义访问模式优化 |
| 扩展方式 | 垂直扩展为主 | 水平扩展为主 |
| 推荐起点 | **默认选择** | 明确的非关系型需求时才选 |

**原则**: 默认选 PostgreSQL。除非有明确的文档型/时序/图数据需求。

### 状态管理选择

| 复杂度 | 推荐方案 | 说明 |
|--------|----------|------|
| 简单 | React state + Context | 组件间少量共享状态 |
| 中等 | Zustand | 轻量级全局状态 |
| 复杂 | Zustand + React Query | 客户端状态 + 服务端状态分离 |
| 企业级 | Redux Toolkit | 需要 middleware、时间旅行调试 |

**原则**: 选择刚好满足需求的最简方案。不要为 5 个全局变量引入 Redux。
