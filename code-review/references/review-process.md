# 代码审查流程

## 审查者心态

好的代码审查就像一位资深导师在指导学员——目标不是挑毛病，而是共同提升代码质量和团队水平。

### 核心原则

1. **建设性优先**：每指出一个问题，必须提供解决方案
2. **解释原因**：不要只说"这样不好"，要说"这样不好，因为..."
3. **提供替代方案**：如果否定现有实现，给出更好的方案
4. **区分必须与建议**：CRITICAL/HIGH 是必须修复的，MEDIUM/LOW 是建议
5. **承认优点**：看到好的代码也要说出来，不要只关注问题

### 避免的行为

- 不要用讽刺或嘲笑的语气
- 不要说"这很明显"或"谁都知道"
- 不要在评论中只写"不对"而不说为什么
- 不要因为个人风格偏好阻碍合并
- 不要一次性提出过多低优先级问题

## 评论格式

### 标准格式

所有审查评论使用统一格式，方便快速识别和处理：

```
[SEVERITY] 问题描述 → 建议修复
```

### 示例

```
[CRITICAL] SQL 查询使用字符串拼接，存在注入风险
→ 改用参数化查询：db.query("SELECT * FROM users WHERE id = $1", [userId])
```

```
[HIGH] async 函数未处理异常，可能导致未捕获的 Promise 拒绝
→ 添加 try-catch 包裹，在 catch 中记录日志并向上抛出
```

```
[MEDIUM] 函数名 `doStuff` 未反映实际行为
→ 建议改为 `calculateOrderTotal`，更清晰地表达函数职责
```

```
[LOW] 导入语句未按约定排序
→ 建议按以下顺序排列：外部库 → 内部模块 → 类型导入
```

### 正面反馈格式

```
[NICE] 这个错误处理模式很好，覆盖了所有边界情况
```

## 审查通过标准

### 通过条件

| 结论 | 条件 | 说明 |
|------|------|------|
| **通过** | 0 CRITICAL + 0 HIGH | 可以直接合并 |
| **有条件通过** | 0 CRITICAL，存在 HIGH | 修复 HIGH 问题后可合并，无需再次审查 |
| **不通过** | 存在 CRITICAL | 必须修复后重新审查 |

### 判断流程

```
发现问题
  ├─ 是否影响安全性或数据完整性？
  │   └─ 是 → CRITICAL（阻碍合并）
  ├─ 是否会导致功能缺陷或潜在 bug？
  │   └─ 是 → HIGH（应该修复）
  ├─ 是否影响代码质量或可维护性？
  │   └─ 是 → MEDIUM（改进建议）
  └─ 仅关乎风格或个人偏好？
      └─ 是 → LOW（可选）
```

## 自审清单（提交审查前）

代码作者在请求审查前应完成以下自检：

### 功能完整性

- [ ] 所有需求已实现并可正常运行
- [ ] 手动测试过主要流程
- [ ] 边界情况已考虑并处理

### 代码质量

- [ ] 无 `console.log` 调试语句
- [ ] 无注释掉的代码块
- [ ] 无硬编码的密钥或凭证
- [ ] 变量和函数命名清晰
- [ ] 文件大小未超过 800 行
- [ ] 函数大小未超过 50 行
- [ ] 嵌套深度未超过 4 层
- [ ] 使用不可变模式（无直接变异）

### 测试

- [ ] 新逻辑有对应测试
- [ ] 测试覆盖正常路径和异常路径
- [ ] 所有测试通过
- [ ] 覆盖率达到 80% 以上

### 安全

- [ ] 用户输入已验证
- [ ] 无 SQL 注入风险
- [ ] 无 XSS 风险
- [ ] 敏感操作有权限检查

### PR 描述

- [ ] PR 标题简洁明了（70 字符以内）
- [ ] 描述中说明了变更的原因（为什么）
- [ ] 描述中包含测试计划
- [ ] 关联了相关 Issue

## 审查周转时间

### 时间期望

| PR 大小 | 期望响应时间 | 说明 |
|---------|-------------|------|
| 小型（< 100 行） | 4 小时内 | 快速审查，通常 15 分钟内完成 |
| 中型（100-400 行） | 1 个工作日内 | 标准审查，预留 30-60 分钟 |
| 大型（400-800 行） | 2 个工作日内 | 深度审查，可能需要多次阅读 |
| 超大型（> 800 行） | 要求拆分 | 不接受审查，要求作者拆分 PR |

### 紧急情况

- 生产环境修复（hotfix）：1 小时内响应
- 安全漏洞修复：立即响应
- 紧急 PR 应在标题中标注 `[URGENT]`

## PR 大小指南

PR 大小直接影响审查质量。就像一顿饭——菜太多反而每道都尝不出味道。

### 大小标准

| 大小 | 变更行数 | 审查质量 | 建议 |
|------|---------|---------|------|
| **理想** | < 200 行 | 最佳 | 推荐 |
| **可接受** | 200-400 行 | 良好 | 标准 |
| **需注意** | 400-800 行 | 一般 | 考虑拆分 |
| **必须拆分** | > 800 行 | 差 | 拒绝审查，要求拆分 |

> 注：行数统计不含自动生成的文件（lock 文件、schema 生成等）

### 拆分策略

当 PR 过大时，按以下方式拆分：

1. **按层次拆分**：数据层 → 业务逻辑层 → 展示层
2. **按功能拆分**：每个独立功能一个 PR
3. **按重构与功能拆分**：先提交重构 PR，再提交功能 PR
4. **按基础设施拆分**：配置/依赖变更与业务代码分开

### 示例

```
# BAD: 一个巨大的 PR
feat: 用户管理模块 (+1200 行)

# GOOD: 拆分为多个 PR
1. feat: 用户数据模型和数据库迁移 (+150 行)
2. feat: 用户 CRUD API 端点 (+200 行)
3. feat: 用户列表页面 (+180 行)
4. feat: 用户编辑表单 (+160 行)
5. test: 用户模块集成测试 (+200 行)
```

## 处理分歧

### 分歧解决流程

代码审查中出现分歧是正常的——就像两位厨师对调味有不同看法，重要的是找到最适合这道菜的方案。

1. **第一轮：评论讨论**
   - 审查者和作者在 PR 评论中交换观点
   - 双方都提供技术依据和参考资料
   - 尝试达成共识

2. **第二轮：同步沟通**
   - 如果评论无法解决，进行实时讨论（会议/电话）
   - 带着具体的代码和数据讨论，不要抽象争论
   - 讨论结果记录在 PR 评论中

3. **第三轮：第三方仲裁**
   - 如果双方仍无法达成一致，邀请第三位团队成员
   - 第三方做出最终决定
   - 所有人尊重最终决定

### 原则

- 对代码不对人——讨论的是代码方案，不是个人能力
- 用数据说话——性能基准、安全标准、业界最佳实践
- 接受"两种方案都可以"的结论——不是所有分歧都有绝对正确答案
- 记录决策原因——方便未来参考

## CODEOWNERS 配置

### 作用

CODEOWNERS 文件确保关键代码变更有对应领域专家审查。就像医院的会诊制度——心脏手术需要心外科专家参与。

### 配置示例

```
# .github/CODEOWNERS

# 全局默认审查者
* @team/core-reviewers

# 前端代码
src/components/ @team/frontend
src/pages/ @team/frontend
src/styles/ @team/frontend

# 后端代码
src/api/ @team/backend
src/services/ @team/backend
src/middleware/ @team/backend

# 数据库相关
src/db/ @team/backend @dba-lead
migrations/ @team/backend @dba-lead

# 基础设施
.github/ @team/devops
docker/ @team/devops
terraform/ @team/devops

# 安全敏感文件
src/auth/ @team/security
src/middleware/auth* @team/security

# 配置文件
*.config.* @team/core-reviewers
package.json @team/core-reviewers
```

### 设置要点

1. 每个关键目录至少指定一个审查者
2. 安全敏感代码需要安全团队审查
3. 数据库变更需要 DBA 审查
4. 基础设施变更需要 DevOps 审查
5. 定期更新 CODEOWNERS，保持与团队结构一致
